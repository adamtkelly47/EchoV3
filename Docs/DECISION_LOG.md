1. **A real mutable-default-style bug** in `core/provenance/records.py` and `core/events/envelope.py`: `record_id: str = new_id("source")` evaluates the default once at class-definition time, so every instance without an explicit id would have received the *same* id. ## 2026-07-20 — Phase 4 (database and repository foundation) completed

Built `infrastructure/database/` (session management + transaction boundaries in `engine.py`, ORM tables in `tables/`, repositories in `repositories/`) and `echo/migrations/` (Alembic, using the async-engine recipe so migrations reuse the app's own asyncpg driver rather than adding a second sync-only driver dependency). Six cross-cutting platform tables, matching Phase 4's explicit list: `audit_events`, `jobs`, `source_records`, `computed_value_records`, `model_calls`, `tool_calls` — all owned by System/Capabilities per `DATA_MODEL.md`, none of them domain-specific (no domain exists yet to own a table).

**The significant real bug this phase**: `asyncpg.connect() got an unexpected keyword argument 'sslmode'`. Neon's connection string carries libpq-style query parameters (`sslmode=require`, `channel_binding=require`); SQLAlchemy's asyncpg dialect forwards URL query parameters verbatim as keyword arguments to `asyncpg.connect()`, which has no `sslmode` parameter (asyncpg uses `ssl=`, passed via `connect_args`). This blocked `alembic revision --autogenerate` on the very first attempt. Fixed in `infrastructure/database/engine.py`: strip query parameters from the URL entirely (`to_asyncpg_url`) and translate `sslmode` into asyncpg's own `ssl=` connect arg explicitly (`wants_ssl`). This is exactly the kind of Neon-specific compatibility detail that would have silently blocked every future phase touching the database if left undiscovered — found by actually running the migration against the real Neon branch, not by reading documentation.

That fix initially only lived in `engine.py`; `migrations/env.py` had its own separately hand-written (and un-fixed) URL-cleaning logic, so the same `sslmode` error reproduced identically on the *second* attempt after the first fix. Rather than patch both copies, `migrations/env.py` was rewritten to call `infrastructure.database.engine.create_engine()` directly — one place owns Neon connection-string handling, not two copies that can silently drift (`CONSTITUTION.md`: Duplicate Business Rules applies to infrastructure code too, not just domain rules).

Verified against the real Neon dev branch end to end, not assumed correct from the code alone:
- `alembic revision --autogenerate` correctly detected all six tables and their indexes from the ORM models.
- `alembic upgrade head` applied cleanly to the empty database.
- `alembic downgrade base` then `upgrade head` again — full reversibility confirmed, not just assumed from the generated downgrade function.
- All six tables' contents queried directly via `information_schema` and confirmed present.
- Integration tests (24 new tests: audit, provenance, jobs, observability repositories, plus two dedicated transaction-boundary tests) ran against the real branch — 63/63 total passing.
- After the full test run, every table was queried again and confirmed to contain zero rows — the test-isolation strategy (uncommitted sessions rolled back on teardown; the one deliberately-committing test cleans up explicitly) leaves no residue on the real dev branch.
- The transaction-boundary tests specifically prove `session_scope`'s contract: one test inserts a row then deliberately raises inside the scope and confirms the row does not exist afterward (rollback); another commits for real, confirms the row exists, then deletes it (commit path, with explicit cleanup since this path doesn't self-discard).
- Confirmed no code in `infrastructure/` or `migrations/` logs anything (grepped for `logger.`/`print(`) and that SQLAlchemy's own SQL-echo logging is never enabled — satisfies Phase 4's "sensitive values are not logged" criterion concretely, not by assertion.
- Rebuilt the live Docker stack (Dockerfile now copies `infrastructure/`, `migrations/`, and `alembic.ini`) and ran `alembic current` inside the *deployed* `backend` container (not the bind-mounted dev container) to confirm the image itself is self-sufficient for running migrations in a real deployment, not just in the verification setup.

## 2026-07-21 — Phase 6 (central approval engine) completed

`domains/approvals/` implements `APPROVAL_MODEL.md`'s state machine exactly (11 states, transitions rejected in code via `VALID_TRANSITIONS`). Unlike Capabilities, this domain owns real persisted state — proposals and approval decisions are domain-owned aggregate roots (CONSTITUTION.md: Aggregate Ownership), so — a deliberate deviation from the Phase 4 pattern — the ORM tables (`ApprovalProposalRow`, `ApprovalDecisionRow`) live inside `domains/approvals/repository.py` itself, not under `infrastructure/database/tables/`, which stays reserved for tables no domain owns. `migrations/env.py` updated to import from both locations so Alembic autogenerate sees every table regardless of which layer it lives in.

All seven of Phase 6's verification criteria are tests, not assertions:
1. **Execution without approval fails** — `execute()` on a merely-submitted (not approved) proposal raises `ApprovalRequiredError`.
2. **Self-approval is impossible** — `approve()` rejects `SYSTEM_ACTOR` as the approving identity with `SelfApprovalNotAllowedError`, enforced in code, not just by convention.
3. **Approval binds to the exact payload** — the `ApprovalDecision` records the payload hash at approval time; a second test tampers with the stored proposal's payload directly (bypassing the intended `edit()` path, to prove the *defense* works even if some future bug skips the front door) and confirms `execute()` still catches the mismatch via `PayloadMismatchError`.
4. **Editing invalidates approval** — `edit()` cancels the old proposal and mints a new `proposal_id`; the new one requires its own fresh approval before it can execute.
5. **Expired approval fails** — using `FakeClock.advance()` past the approval TTL, `execute()` raises `ApprovalExpiredError`.
6. **Duplicate execution is prevented** — calling `execute()` twice on the same already-executed proposal returns the existing result idempotently; the fake write adapter's call count proves it never runs twice.
7. **Failed verification produces the correct state** — a verifier configured to return `False` drives the proposal to `verification_failed` (not silently treated as success), and `execute()` raises `ExecutionUncertainError` rather than returning normally.

A fake external write adapter (`FakeWriteAdapter`) is used per Phase 6's explicit instruction, configurable to succeed or raise, so both the execution-failure and success paths are tested without any real provider integration existing yet.

Real defects found and fixed during verification, not assumed correct: (1) the live Docker stack had silently gone stale since Phase 4 — the Dockerfile wasn't updated to `COPY domains` when Phase 5 introduced it, so the deployed containers had been running Phase-4-era code through all of Phase 5 and into Phase 6's start; caught by remembering to actually rebuild and check `alembic current` in the *deployed* container, not just the dev verification container. (2) `vulture` flagged `ExecutionVerifier.verify`'s parameter as an unused variable — a real false positive, since it's a `Protocol` stub body vulture doesn't understand is meant to be unimplemented; fixed by making the parameter positional-only (`/`) and underscore-prefixed, vulture's own documented convention for "intentionally unused," rather than suppressing the tool or leaving the gap unexplained.

Final state: `ruff`, `mypy --strict`, both custom checkers, `vulture`, `bandit`, `pip-audit` all clean across `apps`+`core`+`infrastructure`+`domains`+`migrations`+`scripts`; 97/97 tests passing (83 carried over + 14 new: 13 unit + 1 integration against real Postgres), 91% overall coverage. Live stack rebuilt, `alembic current` confirmed at head in the deployed backend container, zero row residue in any table after the full test run.

That gap is real and stays open until a key is provided; it is not glossed over as "tested."

Real defects found and fixed:
1. **A genuine mypy strict-mode finding**, not a false positive: `block.text for block in message.content if getattr(block, "type", None) == "text"` — the `getattr` check doesn't type-narrow for mypy, since `message.content` is a large union of Anthropic content-block types and only `TextBlock` has `.text`. Fixed with `isinstance(block, TextBlock)`, which does narrow, and updated the corresponding test fixture from a `SimpleNamespace` stand-in (which can't satisfy `isinstance`) to a real `TextBlock` instance.
2. **The exact same coverage-tooling gap as Phase 5, recurring** — `providers/` existed but wasn't added to `pyproject.toml`'s coverage/vulture/bandit paths, the CI workflow, or `TESTING.md`'s runbook, despite Phase 5's own decision log entry explicitly flagging this as a checklist item for "whenever a new top-level package is introduced." Fixed the same way, and this note exists so a third recurrence is less likely — the actual fix is to treat "add the new top-level package to every tool path list" as a standing step in the phase-completion checklist, not just something to remember after the fact.
3. **The live Docker stack rebuild step got interrupted mid-command** (the background build task was marked "killed" partway through, though the image build itself had already completed by that point) — caught by checking the actual task output rather than trusting the status label, then running `docker compose up -d` separately to bring the already-built images up.

Added a new architecture rule, `providers-must-not-import-domains` (the reverse of Phase 3's `domains-must-not-import-providers`), giving `check_architecture.py` a structural proof for Phase 7's "model calls cannot directly execute external actions" verification criterion: a model adapter has no import path to any domain's execution logic, so it is architecturally incapable of calling `domains.approvals` or `domains.capabilities` directly, regardless of what a model itself might request in its own output text.

Final state: `ruff`, `mypy --strict`, both custom checkers, `vulture`, `bandit`, `pip-audit` all clean across `apps`+`core`+`infrastructure`+`domains`+`providers`+`migrations`+`scripts`; 120/120 tests passing (97 carried over + 23 new, including one live Ollama call), 92% overall coverage. Live stack rebuilt and confirmed healthy with `providers/` included in the deployed image.

## 2026-07-21 — Phase 8 (conversation loop and provenance grounding) completed

Built `domains/conversation/` (sessions/messages, domain-owned persistence per the Approvals precedent), the first real capability (`application/capabilities/current_time.py`), and `application/orchestrators/conversation.py` — a minimal Request Lifecycle: user message -> Capability Planner (structured yes/no model call: "does this need the current time?") -> Capability Executor -> evidence -> Response Generator -> persistence. Wired through `apps/api/routes/conversations.py` (start/send/send-stream/history) and a minimal `frontend/app/chat/page.tsx`.

This phase's real risk was never the plumbing — capability execution, evidence gathering, and persistence all worked the first time and stayed working, confirmed by both fakes-based unit tests and live calls. The risk PROMPT.md actually cares about here is the one CONSTITUTION.md calls out by name: a language model asked "what time is it" defaulting to its own training-data guess instead of the system clock it was just handed. Proving that required live testing against the real local model, not just fakes — and live testing surfaced two genuine, unrelated defects that fakes-based tests could not have caught, because both are properties of what `llama3.2:1b` actually does with specific wording, not of the orchestration code around it.

**Defect 1 — the evidence-grounded response refused to answer.** Given `_RESPONSE_PROMPT_WITH_TIME`'s original wording ("The current UTC time, from the system clock (not your own knowledge), is {ts}... Respond to: {message}"), the live API consistently returned "I can't provide real-time information, including the current date and time" — ignoring the timestamp it had just been given. An isolated manual call to the same adapter with what looked like the same prompt had *succeeded*, which was the misleading part: at default (randomized) sampling, the correct answer was just one of several samples the model could produce, not what it reliably produces. Lowering `temperature` did not fix this — it made it worse, converging deterministically on the refusal. That result was the actual diagnosis: the refusal was the model's single highest-probability completion for that wording, at any temperature; higher temperature's occasional correct answer was luck, not a signal that the prompt was fine. The fix was rewording, not resampling: reframing the injected fact as something the model already has ("[SYSTEM FACT] The current date and time is {ts}... Do not say you lack real-time access — you have just been given the real-time value above") rather than something being relayed to it from a live lookup, which is what had been triggering the model's trained "I don't have real-time access" refusal reflex. Verified deterministically correct (`temperature=0.0`) across three different message phrasings, 3 runs each (9/9), then confirmed live through the actual API repeatedly.

**Defect 2 — the Capability Planner always said yes.** Found while spot-checking a non-time question ("tell me a joke about cats") through the live API and noticing it got current-time evidence injected anyway. Direct testing of `_PLANNER_PROMPT` against `llama3.2:1b` showed it was not discriminating on message content at all: `{"needs_current_time": true}` for every message tested, including "what is 2+2" and "write me a haiku," deterministically, at `temperature=0.0`. This is a materially different failure than Defect 1 — not a refusal reflex, but the model not attending to the actual classification task at 1B-parameter scale from a zero-shot instruction alone. Fixed by rewriting the prompt as a five-example few-shot classifier rather than a bare instruction; re-tested against 11 messages across both categories (jokes, arithmetic, haiku, cooking, literature vs. clock/date/calendar/market-hours questions) and got the correct classification every time. This is the kind of small-model reliability gap CONSTITUTION.md's escalation policy exists for — Phase 8 does not add escalation-on-low-confidence itself, but the finding is recorded here as the concrete motivating case for it.

Neither defect was hypothetical or found by inspection — both were reproduced through the real running stack (`docker compose`, real Ollama, real Postgres, real HTTP calls), root-caused with direct adapter calls that isolated the variable in question (wording vs. sampling for Defect 1; prompt structure vs. model capability for Defect 2), and re-verified the same way after the fix. `ModelRequest` gained an optional `temperature` field (both adapters honor it; `None` preserves each provider's own default) as part of Defect 1's fix — it is used narrowly, only for the planner's classification call (`0.0`) and the evidence-grounded response call (`0.1`, not `0.0`, since determinism converging on a *wrong* answer is exactly as bad as never being deterministic at all — low-but-nonzero was verified, not assumed, to still land on the correct answer across repeated runs).

Verification, all six PROMPT.md Phase 8 criteria, confirmed live against the actual running stack: (1) user asks for the current time via the real HTTP API; (2) the real `system.current_time` capability fires (confirmed via the `evidence` field, not inferred); (3) the response states the exact injected timestamp, not a paraphrase or a guess; (4) the model never falls back to its own training-data date, and — the defect this phase actually found — never refuses either; (5) `GET /conversations/{id}/messages` returns full, correctly-ordered history; (6) `docker compose restart backend` does not lose conversation data (checked message count before and after: 6 and 6).

Final state: `ruff format --check`, `ruff check`, `mypy` (0 issues across 90 source files), `check_architecture.py`, `check_size_limits.py`, `vulture`, `bandit` all clean across `apps`+`core`+`infrastructure`+`domains`+`providers`+`application`+`migrations`+`scripts`; 134/134 tests passing (120 carried over + 14 new, including two live-Ollama-backed integration tests), 90% overall coverage. Live stack rebuilt twice (once per defect fix) and re-verified against the running containers each time, not just against the test suite.

## 2026-07-21 — Phase 9 (Memory foundation) completed

Built `domains/memory/` (the CANDIDATE -> CONFIRMED -> {SUPERSEDED | EXPIRED | DELETED} lifecycle from CONSTITUTION.md's Memory Lifecycle diagram, condensed to the states Phase 9 actually needs — "Review" and "Usage" are policy/consumption concepts, not separate persisted statuses, so they aren't given their own enum members per No Future Scaffolding), plus `application/orchestrators/memory_extraction.py` (extraction using the Model Gateway, same Application-layer placement as Phase 8's conversation orchestrator, for the same reason: cross-domain coordination belongs there, not inside a domain).

**Research before implementation.** DOMAIN_OWNERSHIP.md's Memory section is the authoritative ownership description (per this project's own earlier resolution: DOMAIN_OWNERSHIP.md wins when it and CONSTITUTION.md's illustrative examples disagree). One real, previously-uncaught inconsistency surfaced doing that research: CONSTITUTION.md's "Example Events" list includes `MemoryCandidateCreated` and `MemorySuperseded`, neither of which appears in DOMAIN_OWNERSHIP.md's authoritative Memory event catalog (`MemoryCreated, MemoryUpdated, MemoryArchived, MemoryDeleted, MemoryPromoted, MemoryConfidenceChanged, MemoryConflictDetected`) or in DOMAIN_EVENTS.md's own catalog row, which matches DOMAIN_OWNERSHIP.md. Per the existing precedent, this is noted as a known, non-blocking documentation inconsistency rather than fixed by editing CONSTITUTION.md's illustrative list — no code depends on it, since **no domain in this codebase publishes events yet** (verified: `core/events/envelope.py` is a wire-format contract with no publisher/subscriber wiring anywhere, and Approvals/Conversation don't publish either). Memory does not publish events in this phase either, for the same No Future Scaffolding reason nothing else does — this is recorded so it isn't mistaken for an oversight later.

**A real, undocumented-until-now scope decision:** DOMAIN_OWNERSHIP.md says "Memory writes follow the Approval Engine when required by constitutional policy," but no passage in CONSTITUTION.md actually specifies which Memory writes that means. Decided *not* to route memory confirmation through the Central Approval Engine this phase — Approvals' own documentation (APPROVAL_MODEL.md) describes its first real users as Calendar (Phase 11) and Email (Phase 21), both external-system side-effecting writes; confirming an internal memory record has no external effect to approve. If a future phase decides memory promotion should require human approval (e.g. before it's used to ground responses), that is a real design question for that phase, not retrofitted here.

**Live-testing found two real, distinct small-model reliability problems** in `application/orchestrators/memory_extraction.py`, on top of Phase 8's already-learned lessons:

1. **A single combined "decide-and-extract" prompt was unusable.** The first design asked `llama3.2:1b` to decide *and* extract a candidate fact in one call. Live testing against seven real messages found it hallucinating a fake low-confidence fact for "What time is it?" and, on "Can you write me a haiku?", echoing one of the prompt's own few-shot examples back verbatim instead of processing the actual message — a different, more severe failure mode than the always-true planner bug Phase 8 found. Fixed by splitting into two calls: a binary few-shot gate ("does this message state a durable fact?", reusing Phase 8's proven Capability Planner pattern) followed by a separate extraction call only when the gate says yes. This measured far more reliably.
2. **The gate's few-shot generalization is narrow, and more examples are not a free improvement.** A live end-to-end test asked "What is the current date and time?" and got a false-positive fact extracted. Adding that exact phrase as a 7th few-shot example didn't fix it in isolation — it measurably broke five of the six previously-correct examples, causing the model to default to "true" almost universally. Swapping the new example in place of an existing one instead fixed the new phrase but broke three different previously-correct cases. This is strong evidence the model is closer to matching against the literal example set than learning the general rule, at this parameter count — a real, live-tested finding, not a hypothesis. Kept the original six-example gate (proven reliable across eleven distinct test messages) rather than chasing full phrasing coverage, and documented the limitation directly in the orchestrator's own docstring and in the test that exercises it, choosing a test phrasing within the model's demonstrated reliable range instead of one at its boundary.

**A tooling mistake, caught and fixed, worth recording so it isn't repeated:** `docker compose cp CONTAINER:SRC DEST` nests `SRC`'s contents inside `DEST` when `DEST` already exists locally, rather than overwriting it in place. Used naively while syncing edited files back from the container mid-phase, this silently left a stale, never-imported `domains/memory/memory/*.py` duplicate on disk (dead weight that could have been committed by accident) and meant a `ruff format` pass appeared not to have taken effect on two test files, because the reformatted copies were never actually written back to their real path. Fixed by always either deleting the destination first or using the `SRC/.` trailing-dot form, and re-verified there was no other stray nested directory anywhere else in the tree before the final verification pass.

Verification, all four PROMPT.md Phase 9 criteria, confirmed both by direct unit tests against `MemoryService` (`tests/unit/domains/memory/test_memory_service.py`, one test per criterion, named after the criterion) and live against the real running API: (1) `POST /memory/extract` only ever creates `CANDIDATE` records — confirming one is a distinct, explicit `POST /memory/{id}/confirm` call; (2) `GET /memory/{id}/conflicts` correctly found the "blue" memory as a conflict of a later "green" memory for the same `subject_key`, both live and in tests; (3) `DELETE /memory/{id}` removed a memory from `GET /memory/search` while it remained visible (with `status=deleted`) in the full `GET /memory` user view — checked live, not just asserted in a fake-backed test; (4) every record's `source_type`/`source_id`/`correlation_id` round-tripped through both the Postgres repository and the live API responses.

Final state: `ruff format --check`, `ruff check`, `mypy` (0 issues across 100 source files), `check_architecture.py`, `check_size_limits.py`, `vulture`, `bandit`, `pip-audit` all clean; 166/166 tests passing (134 carried over + 32 new, including three live-Ollama-backed extraction tests and Postgres-backed repository tests), 91% overall coverage, `domains/memory/` itself at 100%. Live stack rebuilt and re-verified against the running containers (full extract -> confirm -> conflict -> search -> delete -> supersede lifecycle exercised over real HTTP, plus a `docker compose restart backend` persistence check), not just against the test suite.
